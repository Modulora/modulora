## Why everything is reviewed

Nothing lists publicly until a curator approves it. This is the trust spine
of the registry: automated checks prove *provenance* (who published, exactly
which bytes install, whether secrets leaked), but a human decides whether a
submission is real, functional, and honestly described. Approval is a
listing decision — it is **not** a code audit, and the evidence panel on
every component says exactly what was and wasn't checked.

## The review standard (`alpha-1`)

Every decision applies a **named, versioned standard**. The current version
is `alpha-1`; the exact checklist below is rendered beside every submission
on the review page, and each check gets an explicit result — *pass*,
*flag*, or *not applicable*:

1. **Publisher & association** — the authenticated publisher's identity is
   scoped correctly; no unauthorized creator, brand, repository, preview, or
   affiliation claim.
2. **Authorization & attribution** — original URL, inspiration,
   fork/derivative, and license/attribution claims are internally
   consistent. Similarity is a review signal, never proof of plagiarism or
   originality.
3. **Listing accuracy** — title, description, taxonomy, framework, demo,
   files, dependencies, and stated behavior agree.
4. **Artifact integrity** — exact files, digest, registry shape, path
   sanitization, and distribution-channel output are present and internally
   consistent.
5. **Install behavior** — install paths copy files only; parity evidence
   and its limitations are shown; unverifiable creator commands stay
   labeled unverifiable.
6. **Risk indicators** — automated secret-scan and other scoped evidence,
   plus obvious executable, network, or data-exfiltration behavior. Not a
   security guarantee.
7. **License & sales scope** — hosted source states its license; external
   sales use a creator-operated destination, and unverified destination
   domains are explicitly disclosed during alpha.
8. **Demo quality & accessibility** — the preview renders the submitted
   component without obvious keyboard, focus, contrast, overflow, or
   misleading-state failures. A listing-quality check, not accessibility
   certification.

**Scope limitation, stated on every record:** approval is a listing
decision against these checks. It is not a security audit, ownership or
originality certification, accessibility certification, or legal review.

## Tool and site usefulness review

External tool and site listings use a separate `tool-alpha-2` standard.
The creator must first prove control of the exact domain through **Settings →
Verified domains**. Curators then inspect an isolated live preview, the
creator's description, category, external pricing label, creator-uploaded site
thumbnails, and server-fetched Open Graph metadata.

The listing decision is based on usefulness rather than metadata differences.
Curators consider whether the site works, helps visitors accomplish a concrete
task or outcome, belongs in Modulora's catalog, is presented accurately by its
copy, pricing label, and thumbnails, and
shows no obvious deception or harmful behavior. Metadata is supporting context.

Approval means the tool was useful and relevant enough for the catalog at the
time of review. It does not certify security, privacy, legality,
accessibility, ownership beyond the scoped domain evidence, or ongoing
availability.

## Similarity screening

Before a submission enters the queue, Modulora screens it against **its own
published release corpus** using exact content digests and normalized token
fingerprints. Exact or high-confidence matches against another creator's
release hold the submission until a curator resolves it; the submitter can
classify each match (original work, authorized republication, derivative,
false positive) with supporting links.

Similarity is a **review signal, never a verdict**: it cannot prove
plagiarism or originality, the screening does not search external sources,
and no listing ever carries an “originality verified” or “plagiarism free”
claim. Scanner failures fail closed — a submission never silently skips
screening into review.

Creators can report plagiarism, impersonation, copied previews, incorrect
attribution, or unauthorized brand association **without a Modulora
account** from any component page. Reports become durable moderation cases
with an append-only history; reporter contact stays private. Cases can lead
to temporary listing restriction, takedown, attribution correction, or
release revocation — each shown publicly in scoped language — and every
action supports appeal.

## Decisions and records

Curators choose **approve**, **request changes**, **reject**, or
**escalate** (which leaves the submission pending for an owner or second
curator). Every outcome — including approval — requires a written
rationale and a complete checklist. Each decision writes an **append-only
review record**: component and version, reviewer, standard version,
per-check results, rationale, and the scope limitation — verbatim, with a
timestamp. Historical records are never rewritten; a resubmitted release
gets a new record.

Creators see the decision and rationale verbatim on their dashboard and by
email, fix, and resubmit. Every new release of an approved component
re-enters the queue.

## What curators can never do

Curation is deliberately separated from money:

- Curators review content; they **cannot** run payout distributions, change
  prices, or see anyone's earnings.
- Approval cannot be bought. Promotion is paid *placement*, marked as such,
  and has no effect on review outcomes or ordering within the queue.
- Plus subscriptions never buy review priority.

## How roles are assigned

- **Curator** — assigned by an owner in **Dashboard → Platform → Admin →
  Roles**, by username. Curators get the Review queue in their dashboard.
- **Owner** — configured only by the `OWNER_USER_IDS` environment variable
  (user IDs, never emails). No code path grants ownership from the UI, so a
  compromised account can't mint owners. Owners run payout distributions
  and assign curators.

Operators self-hosting the registry set `OWNER_USER_IDS` at deploy time;
everything else is managed from the Admin page.
